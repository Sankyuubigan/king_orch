# 📜 Правила проекта King Orch

## 1. Архитектура

### Стек
- **Backend**: Rust (Tauri 2.0) — ядро, LLM-движок, оркестратор, MCP-клиенты
- **Frontend**: TypeScript + Vite — UI, рендеринг чата
- **Модели**: только локальные `.gguf` файлы (llama.cpp через llama-cpp-2)
- **Агенты**: Markdown файлы (`.md`) с YAML frontmatter

### Принципы
- **Изоляция контекста**: каждый агент видит только то, что ему нужно (router — статусы, worker — полные данные)
- **Внешняя память**: [СОСТОЯНИЕ СЕССИИ] (dossier) — KV-хранилище, автоматически передаётся между агентами
- **Без API-ключей**: всё работает локально, никаких внешних LLM-провайдеров

## 2. Система агентов

### Режимы (`mode`)
- `primary` — лицо системы, общается с пользователем. Только эти агенты видны в UI
- `router` — маршрутизатор, ВСЕГДА отвечает JSON с `target`. Видит только статусы досье (✅/❌)
- `worker` — специалист, отвечает текстом. Его ответ автоматически сохраняется в dossier под его ID

### Вызов сабагентов
Агент с `subagents` вызывает их через JSON:
```json
{"thought": "...", "target": "id_сабагента", "task_or_response": "задача"}
```
Возврат: `{"thought": "...", "target": "reply", "task_or_response": "ответ"}`

### Вызов инструментов (MCP)
```json
{"thought": "...", "tool": "имя_инструмента", "arguments": {...}}
```

### Правило: JSON без target/tool — это неполный ответ
Если модель вернула JSON с `thought`, но без `target` или `tool` — это баг генерации. Оркестратор должен перезапросить модель, а не отдавать сырой JSON пользователю.

## 3. Код-стайл

### Rust (backend)
- Функции не длиннее 50 строк. Если длиннее — декомпозируй
- Все `unwrap()` вне тестов — потенциальный баг. Используй `?` или `map_err`
- Логируй ключевые решения: запуск агента, вызов сабагента, вызов инструмента, неполный ответ
- Импорты: `use crate::module::{Item1, Item2};` — явно перечисляем

### TypeScript (frontend)
- Строгая типизация: никаких `any` без веской причины
- Сервисы разделены по SRP: `ChatService`, `ModelService`, `SessionService`
- UI-компоненты в `src/ui/`, утилиты в `src/utils/`

### Markdown (агенты)
- Frontmatter YAML между `---`
- Системный промпт — после второго `---`
- Include-макрос: `<<INCLUDE: database/file.md>>` (путь относительно папки агента)

## 4. Протокол обработки ошибок

### Правило «Наблюдаемость»
Каждый неочевидный путь кода должен логироваться. Если модель вернула неожиданный формат — это должно быть в логах с ⚠️.

### Правило «Лечим причину»
Запрещено глушить ошибки пустыми `catch`, `if (!data) return`, `unwrap()` без обработки. Если данные сломаны — ищи почему, а не прячь.

### Правило «Нет фантазиям»
Если данных недостаточно для ответа — агент обязан сообщить об этом, а не придумывать.

## 5. Реальное время (Real-time events)

### События Tauri → Frontend
- `log` — лог-сообщения системы (отображаются во вкладке Логи)
- `status` — статус обработки + прогресс (отображается в прогресс-баре)
- `progress` — процент прогресса (0-100)
- `agent_thought` — мысли агентов в реальном времени (отображаются в чате как 🧠)
- `subcall_done` — отчёт сабагента в реальном времени (кнопка 📊 в чате)
- `download_progress` — прогресс скачивания модели

### Правило: не дублировать UI-элементы
Если сабагент-отчёт уже отображён через `subcall_done` событие — не рендерить его повторно из финального ответа. Но данные ВСЕГДА сохраняются в `globalChatHistory` для персистентности сессии.

## 6. Сессии и хранение

- Сессии хранятся как JSON файлы в `app_data_dir/sessions/`
- Dossier (`HashMap<String, String>`) — внешняя память сессии
- Ключ `user_query` — оригинальный запрос пользователя (заполняется автоматически)
- Ключи вида `agent_id` — ответы worker-агентов (заполняются автоматически)
- Draft — черновик текста в поле ввода (автосохранение с debounce 500ms)

## 7. MCP-серверы

- Располагаются в `src-tauri/mcp_servers/`
- Формат: `.cjs` (Node.js) или `.ts` (Deno)
- Протокол: JSON-RPC 2.0 через stdin/stdout
- Базовый фреймворк: `mcp_base.cjs` — устраняет дублирование протокола
- Запуск через сайдкары: `bin/node-{target}.exe` или `bin/deno-{target}.exe`

## 8. Сборка и релиз

- `build.bat` — локальная сборка (без подписи если нет ключа)
- `release.bat` — релизная сборка с подписью и публикацией на GitHub
- Сайдкары (yt-dlp, node, deno) скачиваются автоматически в `src-tauri/bin/`
- Версия: `YY.MM.PATCH` (автоинкремент при релизе)

## 9. Ограничения

- Максимальная глубина вложенности сабагентов: 5
- Максимальное количество итераций агента: 30
- Лимит неудачных вызовов инструментов подряд: 3
- Размер файла для AST-карты: до 5MB
- Сайдкары поддерживают только Windows (x86_64-pc-windows-msvc)
